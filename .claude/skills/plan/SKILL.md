# 実装計画（Research-First）

実装に着手する前に、調査 → 計画 → 確認のサイクルを回す。

## フェーズ 0: 対象チケットの特定

対象チケットが明示されている場合はそのチケットを使う。

「ロードマップに従って進めて」「次のチケットをやって」等の指示の場合:

1. `docs/roadmap.md` を読み、現在のフェーズと進捗を把握する
2. `tickets/` 内の未着手チケットと `tickets/done/` の完了チケットを照合する
3. ロードマップの順序と依存関係に基づき、次に対応すべきチケットを特定する
4. 対象チケットをユーザーに提示し、確認を取る

## フェーズ 1: 調査

researcher サブエージェント（**model: haiku**）を使い、以下を調査する:

- 変更対象のコード・ファイルの現状
- 既存の実装パターン・規約
- 影響を受ける可能性のある周辺コード
- 関連するテストの有無と構成

調査範囲が広い場合は複数回 researcher を起動して分割調査する。

## フェーズ 2: 計画作成

planner サブエージェント（**model: opus**）に対象チケットと調査結果サマリを渡し、実装計画を作成させる。

planner には以下を渡す:

- 対象チケットのファイルパスまたは内容
- フェーズ 1 で実施した調査結果のサマリ（researcher の出力をそのまま渡してよい）
- ユーザーから提示された制約・要望（あれば）

planner は以下を含む計画を返す:

- 変更するファイルの一覧と変更概要
- 実装の順序（依存関係を考慮、コミット粒度の提案を含む）
- 設計判断（必要な場合）
- テスト方針（新規テスト・既存テストの修正）
- リスク・懸念点
- **実行モデル推奨**: `sonnet` または `opus`（下記基準で判断）
  - `sonnet`: 計画が明確で定型的な実装（CRUD追加、UI変更、テスト追加など）
  - `opus`: アーキテクチャ変更を伴う、複数システムにまたがる、高度な設計判断が必要な実装

軽微な変更の場合、planner を経由せずメインセッションで直接計画を立ててもよい（判断はオーケストレーターに委ねる）。

## フェーズ 3: ユーザー確認

計画をユーザーに提示し、承認を得る。
修正の要望があれば計画を調整する。

## フェーズ 4: 実行

承認後、以下のいずれかで実装を進める:

- **メインセッションで直接実装**: 小規模な変更、対話しながら進めたい場合
- **developer サブエージェントに委譲**: 計画が明確で自律実行に適する場合。チケットファイルパスと計画内容を渡す
  - developer 起動時は planner が推奨した `model`（`sonnet` または `opus`）を指定する

複数の独立したタスクがある場合は、developer を複数起動して並列実行できる。

## フェーズ 5: 確認

実装完了後、必要に応じて reviewer サブエージェントでレビューを実施する。
