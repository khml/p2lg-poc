# プロジェクト初期化

このスキルはプロジェクト開始時に一度だけ実行する。

## ステップ 1: ヒアリング

ユーザーに以下を質問する。一度に全部聞かず、会話の流れで自然に確認する。

- プロジェクトの目的・概要
- 主要な技術スタック（言語、フレームワーク、DB、インフラ等）
- ディレクトリ構成（既にコードがある場合はコードベースを調査して把握する）
- ビルド・テスト・実行のコマンド
- コーディング規約やスタイルルール（デフォルトと異なるもののみ）
- 開発環境の注意点（必要な環境変数、特殊なセットアップ等）

## ステップ 2: CLAUDE.md の生成

ヒアリング結果に基づいて CLAUDE.md を更新する。以下の構成を守る:

```
# <プロジェクト名>

## 概要
<プロジェクトの目的・説明（2-3行）>

## 技術スタック
<テーブル形式で主要技術を列挙>

## コマンド
<ビルド、テスト、実行、リント等の主要コマンド>

## Git 運用
- commit のみ許可（push はユーザーの明示的な許可が必要）
- commit メッセージは日本語
- 1 commit = 1 つの論理的な変更

## チケット運用
- `tickets/` にテキストファイルで管理（`<4桁番号>_<チケット名>.txt`）
- 最新番号: `tickets/.latest_ticket_number`
- 「チケットステータス」セクションはユーザーのみ更新可能
- `tickets/done/` 内は対応不要
- 詳細: @.claude/rules/ticket-workflow.md

## レポート
- 作業完了時に `reports/` へ簡易レポートを出力
- reports/ は git に commit しない

## 言語
- ユーザーとのやり取り・ドキュメント・コメントは日本語
```

CLAUDE.md は 80 行以内に収める。プロジェクト固有の詳細ルール（コーディング規約等）がある場合は `.claude/rules/` に別ファイルとして作成し、CLAUDE.md からは `@` で参照する。

## ステップ 3: ディレクトリ初期化

以下が存在しない場合は作成する:

- `tickets/` ディレクトリ
- `tickets/done/` ディレクトリ
- `tickets/.latest_ticket_number`（`0000` で初期化）
- `reports/` ディレクトリ
- `.gitignore` に `reports/` が含まれていなければ追加

## ステップ 4: ロードマップ作成

`docs/roadmap.md` が存在しない場合、ヒアリング内容に基づいて雛形を作成する。
存在する場合はスキップする。

## ステップ 5: settings.json の調整

ヒアリングで判明したビルド・テストコマンドを `.claude/settings.json` の `permissions.allow` に追加するか、ユーザーに確認する。

## ステップ 6: 追加エージェント・ルールの提案

プロジェクトの特性に応じて、追加のエージェントやルールが必要か検討しユーザーに提案する。

例:
- インフラ管理が必要 → インフラ専用エージェントの追加
- 特定のフレームワーク規約がある → `.claude/rules/` に専用ルールファイルの追加
- E2E テストが必要 → テスト専用エージェントの追加

ユーザーの承認を得てから作成する。

## ステップ 7: 完了確認

セットアップ内容のサマリーをユーザーに提示し、問題ないか確認する。
変更を commit する。
