# チケット運用ルール

## チケット管理

- 全タスクは `tickets/` ディレクトリにテキストファイルとして管理する
- ファイル名形式: `<4桁チケット番号>_<チケット名>.txt`
- 最新チケット番号は `tickets/.latest_ticket_number` に記載されている
- チケット新規作成時はこの番号を参照し、作成後に更新する

## チケットステータス

- チケット内の「チケットステータス」セクションはユーザーのみが更新する。編集禁止
- 完了したチケットはユーザーが `tickets/done/` に移動する
- `tickets/done/` 内のチケットは内容に関わらず対応不要

## ロードマップとの連携

- `docs/roadmap.md` にチケットの実行順序・依存関係が定義されている
- ロードマップに従って作業を進める場合は、ロードマップを読んで次の対象チケットを特定する
- チケット間の依存関係がある場合は、依存元を先に完了させる

## チケット対応時のルール

### 作業の進め方

1. チケットファイルを読み、対応内容と受け入れ条件を把握する
2. 関連ドキュメント・コードを確認する
3. 修正作業の前にテストを実行する（既存の不具合と修正による不具合を区別するため）
4. 実装を進める。疑問点・不明点がある場合は推測して続行する
5. 実装後にテストを実行して確認する
6. 適切な粒度でコミットする
7. 簡易レポートを出力する

### レポート

- 作業完了時に `reports/` ディレクトリへレポートを出力する
- ファイル名: `<4桁チケット番号>_<チケット名>_<YYYYMMDD>_report.txt`
- reports/ は git に commit しない
- 簡潔に書く。トークン消費を最小限に抑える

### レポートフォーマット

```
チケット: <番号> <タイトル>
日付: <YYYY-MM-DD>

## やったこと
- <実施内容を箇条書き>

## コミット
- <ハッシュ>: <メッセージ>

## 推測した点
<推測して判断した事項があれば記載。なければ省略>

## 発見事項
<チケット範囲外の発見があれば記載。なければ省略>
```

### チケット範囲外の発見事項

- チケット対応中にチケット範囲外の不具合や改善点を発見した場合、レポートに記載する
- 対応完了後にユーザーに報告する

## チケットフォーマット

新規チケット作成時は以下のフォーマットに従う:

```
チケット番号: <4桁番号>
タイトル: <チケットタイトル>

---

## 概要
<このチケットで何を行うかの概要>

## 背景
<なぜこのチケットが必要なのか>

## 対応内容
- <具体的な対応項目>

## 優先度
<高/中/低>

## 依存関係
- <依存するチケットがあれば記載>

---

## チケットステータス
未着手
```
