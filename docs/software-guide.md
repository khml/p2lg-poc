## そもそも監視とは？なぜ必要なのか

Webアプリケーションを本番環境で動かしていると、こんな疑問が生まれます:

- **「今、サーバーは正常に動いているか？」**
- **「レスポンスが遅くなっていないか？」**
- **「エラーが増えていないか？何が起きたのか？」**

これらに答えるために「監視」が必要です。監視には大きく2つの柱があります:

| 種類 | 何を見るか | 例 |
|---|---|---|
| **メトリクス**（数値） | システムの「健康状態」を数値で継続的に記録する | リクエスト数、レイテンシ（応答時間）、エラー率 |
| **ログ**（テキスト） | 「何が起きたか」を時系列の記録として残す | `ERROR: DB接続失敗`、`INFO: リクエスト受信` |

メトリクスは「体温計で熱を測る」イメージ、ログは「日記に出来事を書き留める」イメージです。  
このハンズオンでは、この2つを収集・保存・可視化するシステムを構築します。

## 全体像：4つのソフトの役割分担

このプロジェクトでは4つのソフトウェアが連携して動きます。

| ソフトウェア | 一言で言うと | 担当する仕事 |
|---|---|---|
| **Prometheus** | メトリクスの収集係 | アプリから数値データを定期的に取りに行く |
| **Loki** | ログの保管係 | ログを受け取って保存・検索できるようにする |
| **Promtail** | ログの配達員 | アプリのログを拾って Loki へ届ける |
| **Grafana** | データの見える化係 | PrometheusとLokiのデータをグラフ・表で表示する |

```
demo-app（サンプルアプリ）
  │
  ├─ /metrics エンドポイント ◄── Prometheus が5秒ごとに取りに来る
  │
  └─ stdout（ログ出力）──► Promtail が読み取り ──► Loki に保存
                                                        │
                                          Grafana が両方のデータを可視化
                                       ↑                ↑
                                   Prometheus          Loki
```

## Prometheus — メトリクスを定期収集する仕組み

### ひとことで言うと

**「決まった時間ごとに、アプリの健康状態を数値で測りに行く収集係」**

### 解決する課題

アプリがメトリクスを持っていても、誰かが「取りに来てくれない」と記録が残りません。  
また、複数のサーバーを監視したいとき、それぞれに個別に接続して確認するのは大変です。

Prometheus がない世界では、こんな困りごとが起きます:
- エラーが増えていても、誰かが手動で確認するまで気づかない
- 「いつからレスポンスが遅くなったのか」を後から調べる手段がない

### 例え話

看護師が「毎朝9時に病室を巡回して、全患者の体温・血圧を測って記録する」イメージです。  
看護師（Prometheus）が能動的に測りに行く（これを **Pull型** と言います）ことで、患者（アプリ）は「測られたとき」だけ数値を返せばよい仕組みになっています。

### ハンズオンでの役割

- サンプルアプリ（demo-app）の `http://app:5000/metrics` に**5秒ごとにアクセス**してメトリクスを収集する
- 収集したデータは Prometheus 内部に保存され、**PromQL**（Prometheusのクエリ言語）で検索・集計できる
- ブラウザで `http://localhost:9090` を開くと、収集したデータをグラフで確認できる

> **PromQL（プロムキューエル）とは?** Prometheusが持つデータを検索・集計するための専用の問い合わせ言語です。例えば `rate(app_request_total[1m])` と書くと「1分あたりのリクエスト数」を計算できます。

## Loki — ログの保存・検索基盤

### ひとことで言うと

**「ラベルを付けてログを整理し、あとから素早く検索できるようにする保管係」**

### 解決する課題

アプリが出力するログは、放っておくとサーバー内のテキストファイルとして散らかります。  
複数のコンテナ・サーバーからのログを一か所で横断検索するには、専用のシステムが必要です。

Loki がない世界では:
- 「5分前のエラーログ」を見るために SSH でサーバーに入り、`grep` コマンドを叩くしかない
- コンテナが消えるとログも消える

### 例え話

「日記をラベル付きで本棚に整理する司書」のイメージです。  
Loki はログの**中身（テキスト）にはインデックスを付けず**、「どのアプリのログか」「どのコンテナか」といった**ラベルだけでインデックスを作る**設計になっています。これにより、大量のログでも軽量に動作します。

### ハンズオンでの役割

- Promtail から送られてきたログを受け取り、保存する（`http://loki:3100` で受信）
- `service="app"` や `container="demo-app"` といったラベルで検索できる状態で管理する
- Grafana から **LogQL**（Loki のクエリ言語）で問い合わせを受け付ける

> **LogQL（ログキューエル）とは?** Lokiでログを検索・集計するための専用言語です。`{service="app"} |= "ERROR"` と書くと「appサービスのERRORを含むログ」を絞り込めます。

---

## Promtail — ログをLokiへ届けるエージェント

### ひとことで言うと

**「コンテナから出てくるログを自動で拾い、Lokiへ届け続ける配達員」**

### 解決する課題

Loki はログを保存できますが、「ログを送り込む仕組み」は別途必要です。  
各アプリが自分でLokiに送信するように改修するのは大変ですし、Dockerのログ出力をそのまま活用したい場面が多くあります。

### 例え話

「各家庭（コンテナ）のポスト（stdout）に投函される手紙（ログ）を毎日集めて、郵便局（Loki）へ届ける郵便配達員」のイメージです。  
アプリ側は「ログをstdoutに出力するだけ」でよく、収集・転送はPromtailが全部やってくれます。

### ハンズオンでの役割

- Docker のソケット（`/var/run/docker.sock`）を通じて、起動中のコンテナのログを**自動検出**する
- 検出したログに `container` や `service` などのラベルを付けてから Loki へ転送する（これを **Push型** と言います）
- 設定ファイル（`promtail/promtail-config.yml`）で「どのコンテナのログを収集するか」を制御できる

---

## Grafana — データを統合可視化するダッシュボード

### ひとことで言うと

**「Prometheusのメトリクスとlokiのログを一画面でグラフ・表として見せる指令室」**

### 解決する課題

PrometheusやLokiにデータが蓄積されても、数値のまま眺めるのは大変です。  
「どのエンドポイントが遅いか」「エラーが増え始めたのはいつか」を**視覚的に**把握するには、グラフや表が必要です。

### 例え話

「車のコクピット（計器パネル）」のイメージです。  
速度計・燃料計・エンジン警告灯といった複数のセンサーデータが一画面にまとまっているように、Grafanaは複数のデータソース（PrometheusやLoki）のデータを一つのダッシュボードに集約して表示します。

### ハンズオンでの役割

- 起動時に Prometheus と Loki のデータソースが**自動で設定**される（`grafana/datasources.yml`）
- 「demo-app 監視ダッシュボード」が**自動でプロビジョニング**される（`grafana/demo-dashboard.json`）
- ブラウザで `http://localhost:3003` を開くとすぐにグラフが見られる（admin/admin でログイン）
- **Explore** 画面から PromQL・LogQL を自由に試すこともできる

## コラム：Pull型 と Push型の違い

このスタックには「Pull型」と「Push型」の2つのデータ収集方式が混在しています。

| 方式 | 説明 | 使われているところ |
|---|---|---|
| **Pull型** | 収集側（Prometheus）が能動的にデータを取りに行く | Prometheus → demo-app |
| **Push型** | 送信側（Promtail）がデータをサーバーへ送り込む | Promtail → Loki |

**なぜPrometheusはPull型なのか?**  
監視対象のアプリが「今正常に動いているか」を確認するには、Prometheus側が定期的にアクセスしてみる（Pull）のが確実です。アプリが落ちていれば「取りに行けない」という形で障害を検知できます。

**なぜPromtail→LokiはPush型なのか?**  
ログは「発生したとき」に即座に転送する方が効率的です。また、Lokiはどのコンテナから送られてくるか事前に知る必要がないため、Push型で柔軟に対応できます。

## コラム：メトリクスとログの違い

| 観点 | メトリクス | ログ |
|---|---|---|
| **形式** | 数値（例: リクエスト数 = 42） | テキスト（例: `ERROR: DB接続タイムアウト`） |
| **向いていること** | 傾向の把握・アラート・グラフ化 | 原因調査・イベント追跡 |
| **保存効率** | 数値なのでコンパクト | テキストなのでデータ量が大きくなりやすい |
| **このスタックの担当** | Prometheus が収集・保存 | Loki が収集・保存 |

「何か異常が起きている」はメトリクスで気づき、「なぜ異常が起きたか」はログで調べる、という使い分けが一般的です。

## 用語ミニ辞典

| 用語 | 説明 |
|---|---|
| **スクレイプ（Scrape）** | Prometheusがメトリクスを取りに行く動作のこと。「掻き集める」が語源 |
| **エクスポーター（Exporter）** | アプリがPrometheusに向けてメトリクスを公開する仕組み。demo-appの `/metrics` がこれに当たる |
| **ラベル（Label）** | メトリクスやログに付けるタグ（例: `status="200"`, `service="app"`）。グループ化・絞り込みに使う |
| **PromQL** | Prometheusのデータを検索・集計するクエリ言語（Prometheus Query Language） |
| **LogQL** | Lokiのログを検索・集計するクエリ言語（Log Query Language）。PromQLに似た文法 |
| **データソース（Data Source）** | Grafanaが接続するデータの取得元。このハンズオンでは Prometheus と Loki の2つ |
| **プロビジョニング（Provisioning）** | 設定ファイルをもとに、起動時に自動でセットアップすること。Grafanaのダッシュボードが自動で表示されるのはこのおかげ |
| **Pull型 / Push型** | データを「取りに行く」か「送り込む」かの方式の違い（上のコラム参照） |
