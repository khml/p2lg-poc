チケット番号: 0005
タイトル: 初心者向けソフトウェア解説ドキュメント作成

---

## 概要

ハンズオンを試す前に読む「各ソフトウェアの入門解説ドキュメント」を新規作成する。
Prometheus / Loki / Promtail / Grafana それぞれについて、目的・解決する課題・役割を
初心者向けに分かりやすく説明する。

対象ファイル: `docs/software-guide.md`（新規）

## 背景

既存ドキュメント:
- `docs/architecture.md` — 技術仕様・設定値（エンジニア向け詳細）
- `docs/handson-guide.md` — 操作手順書（操作は分かるが背景知識の説明なし）

ハンズオンを試す前に「各ソフトが何者で、なぜ必要か」を理解できる入口ドキュメントが存在しないため作成する。

## 対応内容

### docs/software-guide.md の新規作成

以下のセクション構成で作成する:

1. **はじめに** — このドキュメントの目的・読む順番の案内
2. **そもそも監視とは？なぜ必要なのか** — メトリクスとログの2本柱の説明
3. **全体像: 4つのソフトがどう連携するか** — 役割分担の一覧表・簡易図
4. **Prometheus** — メトリクスを定期収集する仕組み。Pull型の概念を噛み砕いて説明
5. **Loki** — ログの保存・検索基盤。ラベルベースのシンプルなインデックスを説明
6. **Promtail** — Docker コンテナのログを Loki へ転送するエージェント
7. **Grafana** — Prometheus・Loki のデータを統合可視化するダッシュボード
8. **コラム: Pull型 と Push型の違い** — データの流れの違いとその理由
9. **コラム: メトリクスとログの違い** — 数値 vs テキスト、用途の使い分け
10. **用語ミニ辞典** — スクレイプ・ラベル・PromQL・LogQL 等を1〜2行で解説
11. **次のステップ** — handson-guide.md への誘導

### 各ソフトのセクション構成（統一フォーマット）

| 小見出し | 内容 |
|---|---|
| ひとことで言うと | 1文の要約 |
| 解決する課題 | このソフトがない世界ではどう困るか |
| 例え話 | 日常的な比喩 |
| ハンズオンでの役割 | p2lg-poc での具体的な動作 |

### 例え話の方針

- Prometheus: 決まった時間に患者の血圧を測りに来る看護師（Pull型）
- Loki: ラベルを付けて日記を本棚にしまう司書
- Promtail: コンテナのログを拾って Loki に届ける郵便配達員
- Grafana: 複数センサーを一枚に映す車のコクピット

### 既存ドキュメントへの導線追加

- `README.md` のドキュメント一覧に `software-guide.md` を追加
- `docs/handson-guide.md` の冒頭に「事前に software-guide.md を読むことを推奨」の一文を追加

## 執筆指針

- 1ソフトあたり 500〜800字を目安にする
- 専門用語を使う場合は必ず一行で補足説明を付ける
- 具体的な設定値（ポート番号等）は最小限にし、詳細は `architecture.md` へリンクで逃がす
- 説明はこのハンズオン環境（p2lg-poc）の文脈と紐付ける

## 品質チェックポイント

- [ ] 専門用語が登場したら、その場で1行補足しているか
- [ ] 各ソフトの説明構成が揃っているか
- [ ] 「なぜ必要か」が各ソフトで明示されているか
- [ ] `architecture.md` と矛盾する記述がないか（ポート・スクレイプ間隔等）
- [ ] README と handson-guide からの導線が追加されているか
